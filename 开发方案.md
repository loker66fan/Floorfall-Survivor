# Floorfall Survivor 技术文档

## 1. 项目定位

`Floorfall Survivor` 是一个“持续向下生存”的 2D 街机小游戏，目标是在不断上移的世界中保持落脚、规避危险平台，并尽可能获得更高分数。

当前实现提供两套运行形态：

- `Pygame` 桌面窗口版
- `Vue 3 + Vite` 网页版

两端共享同一套核心玩法设计：

- 玩家左右移动
- 重力下落
- 平台落脚判定
- 世界向上滚动，营造持续下坠感
- 多种平台机制
- 时间与距离混合计分
- `easy` / `hard` 难度模式

## 3. 启动流程

### 3.1 统一入口

`main.py` 是统一入口，负责解析命令行参数：

- `--mode prompt|desktop|web`
- `--difficulty easy|hard`
- `--no-browser`

### 3.2 启动器

`launcher.py` 提供两类启动方式：

- 图形启动器：基于 `tkinter`
- 命令行回退：当图形环境不可用时使用

图形启动器支持：

- 选择 `Pygame` 窗口版
- 选择 `Vue` 网页版
- 选择 `easy` / `hard` 难度

### 3.3 网页版服务

网页模式使用 Python 标准库的 `ThreadingHTTPServer` + `SimpleHTTPRequestHandler` 在本地启动静态服务。

`launcher.py` 中的 `SpaRequestHandler` 做了单页应用兼容：

- 静态文件存在时按文件返回
- 否则回退到 `index.html`

浏览器地址会附带 `?difficulty=easy|hard` 查询参数，供前端读取。

## 4. 游戏状态机

当前状态定义在 `settings.py`：

- `STATE_START`
- `STATE_PLAYING`
- `STATE_PAUSED`
- `STATE_GAME_OVER`

状态流转规则：

- `start -> playing`：按 `Space / Enter`
- `playing -> paused`：按 `P / ESC`
- `paused -> playing`：按 `P / ESC` 或 `Space / Enter`
- `playing -> game_over`：玩家死亡或掉出底部
- `game_over -> playing`：按 `Space / Enter`

## 5. 玩法核心

### 5.1 玩家系统

`player.py` / `web/src/game/engine.js`

玩家属性包括：

- 位置 `x / y`
- 前一帧位置
- 垂直速度
- 朝向
- 存活状态

玩家行为包括：

- 左右移动
- 重力下落
- 落到平台时停住
- 在弹跳平台上反弹
- 撞到危险平台死亡

### 5.2 世界滚动

世界并不是让玩家持续下落到无穷空间，而是通过“平台整体向上移动”模拟持续下坠。

在 `game.py` 与 `engine.js` 中：

- 根据时间提升滚动速度
- 计算本帧滚动量 `scroll_step`
- 所有平台统一执行 `shift(-scroll_step)`

### 5.3 碰撞判定

落地判定只在玩家处于下落阶段时生效：

- 玩家当前竖直速度必须向下
- 玩家矩形与平台矩形重叠
- 上一帧底部位于平台顶部附近

危险判定：

- 尖刺平台碰撞立即死亡
- 顶部边界挤压到实体平台也会死亡

## 6. 平台系统

### 6.1 平台类型

平台类型常量：

- `PLATFORM_NORMAL`
- `PLATFORM_MOVING`
- `PLATFORM_FRAGILE`
- `PLATFORM_SPIKE`
- `PLATFORM_BOUNCE`

### 6.2 行为差异

普通平台：

- 可稳定站立

移动平台：

- 左右往返移动
- 可带动玩家横向位移

脆弱平台：

- 第一次踩中后进入破碎倒计时
- 倒计时结束后移除

尖刺平台：

- 视为危险平台
- 接触后立即死亡

弹跳平台：

- 触发更强反弹

## 7. 平台生成算法

`level_manager.py` 和 `web/src/game/engine.js` 中各维护了一套等价逻辑。

生成策略包括：

- 初始安全平台
- 固定活动平台目标数量
- 根据玩家当前位置上下文生成下一批平台
- 平台宽度、横向可达距离、平台间距、平台类型概率受难度影响

### 7.1 关键生成参数

基础参数来自：

- `PLATFORM_MIN_WIDTH`
- `PLATFORM_MAX_WIDTH`
- `PLATFORM_GAP_MIN`
- `PLATFORM_GAP_MAX`
- `PLATFORM_TARGET_COUNT`
- `PLATFORM_SPAWN_BUFFER`

### 7.2 难度修正

`settings.py` 与 `web/src/game/constants.js` 中定义了两组难度预设：

- `easy`
- `hard`

每组预设可影响：

- 世界滚动速度倍率
- 平台最小/最大间距偏移
- 平台宽度偏移
- 活动平台目标数量
- 出生缓冲区大小
- 横向可达范围
- 安全平台倾向
- 危险平台权重
- 移动/脆弱/弹跳平台权重

### 7.3 简单模式

特点：

- 平台更密
- 平台更宽
- 安全平台出现概率更高
- 危险平台权重更低
- 世界滚动更慢

### 7.4 困难模式

特点：

- 平台更稀
- 平台更窄
- 危险平台与复杂平台更常见
- 世界滚动更快

## 8. 分数系统

分数由两个维度组成：

- 生存时间
- 累计滚动距离

Python 侧：

```python
self.score = int(self.elapsed * SCORE_TIME_FACTOR + self.total_scroll * SCORE_DISTANCE_FACTOR)
```

网页端保持相同公式。

最高分存储：

- 桌面版：`best_score.txt`
- 网页版：`localStorage`

## 10. 网页构建流程

开发依赖：

- `vue`
- `vite`
- `@vitejs/plugin-vue`

构建方式：

```bash
python3 build_web.py
```

构建脚本流程：

1. 复制 `web/` 到 `/tmp` 临时目录
2. 在临时目录执行 `npm install`
3. 执行 `npm run build`
4. 把生成的 `dist/` 复制回项目 `web/dist/`

这样做是为了降低当前工作目录下本地二进制执行限制带来的影响。

## 11. 运行与发布建议

### 本地运行

- 桌面版：适合直接调试手感与碰撞
- 网页版：适合移动端和浏览器交互验证

### 发布方向

后续可以扩展为：

- 在线排行榜
- 音效与粒子特效
- 多角色皮肤
- 打包为桌面发行版
- 将网页版部署到静态站点

## 12. 当前限制

- Python 与 Vue 各自维护一套逻辑，当前以“行为一致”为目标，尚未抽取成真正共享的核心引擎
- 网页版最高分使用浏览器本地存储，未与桌面版同步
- 暂无音效、存档槽、排行榜和联网功能

## 13. 推荐维护方向

- 提取共享玩法参数，减少双端维护成本
- 为关键玩法增加自动化回归测试
- 增加资源目录规范，如音频、图像、字体
- 增加部署脚本与 GitHub Actions
